# Cognitive Debt

This is loaded **always** at Step 6 (self-critique) and is the most underrated risk of 2026 software architecture.

The phenomenon: as AI generates more of the code, humans develop less of the mental model needed to understand, debug, and evolve it. Tech Radar Vol 34 names this as a foundational concern for the industry — not because AI-assisted development is bad, but because *unmanaged* AI-assisted development creates a debt that compounds.

The architect's job is to design systems that humans can still reason about even when most of the code was written by an agent.

## What cognitive debt is

It's not technical debt. Technical debt is "we know this code is sub-optimal; we'll fix it later." Cognitive debt is "we don't know how this code works because we didn't write it carefully — and the person who could explain it is the agent that already moved on."

Symptoms:

- A feature was shipped fast, but no human on the team can confidently explain how it works end-to-end
- When a bug is reported, debugging starts by asking the agent to investigate (because the human doesn't know where to look)
- New engineers can't onboard onto modules that "the agent built last quarter"
- The team is afraid to refactor because nobody knows what depends on what
- Architecture diagrams are stale because they were never built; the diagram lives in scattered code the agent wrote
- The same pattern is implemented differently in three places because the agent generated each one fresh

These are not problems you discover in week 1. They emerge in months 6-18, exactly when the system needs to evolve fastest.

## Why it's worse with AI than without

**Pre-AI**: a developer writes code. They understand it because they wrote it. They may not document it well, but they remember the why. The team knowledge is in the heads.

**With AI**: an agent writes code. The developer reviews and merges. They may understand parts of it; they probably don't understand all of it. They didn't make every micro-decision. The why was in the agent's transient context, which is now gone.

Multiply this by hundreds of merges. The system has been *built* but not *understood*.

The compound effect: the next time something needs to change in that area, the human's instinct is to ask the agent to do it again — because re-learning the code costs more than re-prompting. But the agent doesn't remember last time either, so it starts fresh, possibly with different conventions. Over time, the system loses internal consistency.

## How architecture can fight cognitive debt

The architect's leverage: design systems whose structure is **legible** — the code shape itself communicates the design intent — and require **artifacts** that compress understanding for humans.

### 1. Boring architecture

The biggest tool against cognitive debt is *boring architecture*: well-known patterns, fewer abstractions, clear boundaries.

A modular monolith with named domains (`hr`, `payroll`, `finance`) is more legible than a microservices mesh with 30 services and ad-hoc communication. Each module has a clear responsibility a human can hold in their head.

When in doubt, choose the architecture that a senior engineer can explain in 5 minutes on a whiteboard. If it takes 30 minutes, the system is over-built for the current problem.

### 2. Conventions, enforced

Naming conventions, file structures, error handling patterns, test patterns — when these are **consistent**, the human reading code generated by an agent can predict where things are and how they work. The agent should follow the conventions; the conventions should be in a CONSTITUTION file or skill the agent loads.

Without conventions, every new piece of code is novel. With conventions, novelty is concentrated in the actual business logic, not in the wiring.

### 3. Architecture decision records (ADRs)

ADRs are the single most effective tool against cognitive debt. A future engineer (or future you) wondering "why did we do it this way?" should find the answer in the ADR log, not have to reverse-engineer from code.

The discipline:

- Every one-way decision gets an ADR
- ADRs are written **at the time** of the decision, not retroactively
- ADRs include the alternatives considered and why they were rejected
- ADRs are kept in the repo, not in a separate wiki that drifts

Without ADRs, every architectural decision is implicit, and every "why" question becomes archaeology.

### 4. Diagrams that are kept current

A C4 Level 1 + Level 2 diagram in the repo, updated when major components change. Mermaid is sufficient. The diagram doesn't need to show every detail; it needs to show the shape of the system.

A new engineer should be able to look at the diagram and understand the major components and how they communicate within 5-10 minutes. If they can't, the diagram is wrong or the architecture is too complex.

### 5. Onboarding test

A periodic test of cognitive health: bring in a fresh pair of eyes (a new hire, an external review, an architect from another team) and have them try to explain a key flow in the system. If they can do it from documentation alone in <30 minutes, the cognitive debt is manageable. If they can't, you've accumulated debt.

This test catches what code metrics miss.

### 6. Pair the agent with a human

The pattern that works: the human is the architect (decides what), the agent is the implementer (does what), and the human reviews not just the code but the **why**. If the human can't explain why the agent's approach is correct, they don't merge — they iterate.

This is slower than "agent does everything." It's also the only way to keep cognitive debt from compounding.

### 7. Skills as compressed knowledge

When a pattern is solved well, capture it as a skill (or rule, or template). The skill becomes the canonical way to do that pattern. Future agent runs reach for the skill instead of re-deriving the solution. Future humans read the skill instead of reading scattered code.

This is the compounding side: each well-captured skill reduces the cognitive load on subsequent work.

## What to look for during Step 6 self-critique

Ask these questions of any architecture spec:

- [ ] **Could a senior engineer with no prior context understand this system from the spec alone in <30 minutes?**
- [ ] **Are the architectural decisions documented as ADRs, or are they implicit in the code?**
- [ ] **Is there a current diagram of the major components and their relationships?**
- [ ] **Are conventions documented (where, how) so the agent and human follow the same patterns?**
- [ ] **Is the design boring (well-known patterns) or novel (custom abstractions)? If novel, is the novelty justified?**
- [ ] **If 80% of the code is agent-generated, what's the human's mental model of the system, and is it captured anywhere?**
- [ ] **What happens when the engineer who shepherded this leaves? How long does onboarding the next person take?**
- [ ] **Are there sections of the system labeled "the agent built this and I'm not sure how it works"? If so, that's a remediation item.**

If any of these is "no", the design is more cognitively expensive than it appears, and that cost is not yet visible — but it will be.

## Concrete patterns that reduce cognitive debt

### Domain-per-module

Whether modular monolith or microservices, organize by domain (`auth`, `billing`, `hr`, `delivery`) not by layer (`controllers`, `services`, `repositories`). When a developer says "I'm working on billing", they go to one place.

### One way to do each common thing

- One auth pattern (not three)
- One way to define an API endpoint (not "depends on the module")
- One way to handle errors (not bespoke per route)
- One pattern for background jobs
- One pattern for database queries

The agent will follow whatever pattern it sees; if the codebase is consistent, the agent stays consistent.

### Public APIs vs internal helpers

When code is consumed across modules, mark it explicitly (a `public` folder, an `index.ts` that re-exports). When it's internal, hide it. This makes the surface area legible.

### Tests as documentation

Tests that read like specs explain what the code is supposed to do. A new engineer reads the test, understands the contract, then reads the implementation. Tests with descriptive names ("should return 403 when user is not in tenant") teach better than comments.

### Limited abstraction layers

Each abstraction layer has a cost: it must be learned, it must be navigated. Three layers of abstraction costs less than seven. A direct approach often beats a "clean architecture" with N concentric circles, especially in a 5-engineer team.

When you're tempted to add an abstraction "for future flexibility", ask: "have we needed this flexibility three times already?" If not, the abstraction is speculative.

### Comments that explain why, not what

`// increments counter` is noise. `// counter is per-tenant; we need this even though it looks redundant because tenant X had a race condition where ...` is irreplaceable.

Code shows what; comments explain why. Future engineers (and agents) need the why.

### CONSTITUTION file (or equivalent)

A repo-level file that documents the invariants: how things are done here, what's not negotiable, what conventions exist. The agent loads this. The human refers to this. The team reviews this.

Examples:
- "The single source of truth for metrics is the ICO Engine. No metrics calculated inline."
- "Every domain is a Postgres schema. No cross-schema joins outside designated views."
- "All connections via `@/lib/db`. No `new Pool()` anywhere else."

The CONSTITUTION is what makes the system *yours* and not generic AI-generated code.

## Questions to ask the agent

When delegating to an implementation agent (Claude Code, Codex), the architect's prompts should include:

- "What's the simplest version of this?"
- "What existing code or skill does this overlap with?"
- "If this fails in production in 6 months, what's the most likely reason?"
- "What's the riskiest part of this design?"
- "What did you skip or leave for later, and why?"
- "Walk me through the data flow end-to-end."

The agent's answers reveal whether the agent (and you) understand the design — or whether you're shipping cognitive debt.

## The 18-month test

The most useful internal benchmark: when a major incident hits at 3am, 18 months from now, can the on-call engineer reason about this system?

If today's design only makes sense to people who were in the room, today, with the agent's transient context, the design has cognitive debt baked in.

If the design is documented, conventional, and legible, the 3am engineer has a fighting chance.

## Recovery: paying down cognitive debt

When you find debt that's already accumulated (in an audit, or reviewing existing code):

1. **Map what's there** — start with a diagram of major components
2. **Pick the highest-risk module** (the one with most unknowns × most importance)
3. **Have a human read the code, write the ADRs retroactively**, document the conventions actually used
4. **Ask: is the existing design defensible, or should it be refactored?**
5. **If refactor: build conventions and tests first, then refactor sections incrementally**

This is slow work. There's no shortcut. The hope is to never let the debt accumulate that much in the first place.
