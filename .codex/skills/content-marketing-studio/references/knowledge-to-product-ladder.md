# Knowledge-to-Product Ladder

Usa este contrato cuando una Pillar o territorio editorial pueda evolucionar hacia un ebook, una tool o el
soporte intelectual de un producto futuro. No lo cargues para atomización cotidiana.

## Modelo

```text
Pillar -> satélites -> ebook/workbook -> tool diagnóstica -> producto
aprender    profundizar    diseñar        diagnosticar       operar
```

Cada capa debe resolver un trabajo distinto:

| Capa | Trabajo | Gate |
|---|---|---|
| Pillar | definir la categoría y el mapa completo | tesis, evidencia, canonical y baseline |
| Satélites | profundizar intenciones independientes | pregunta real, aporte propio, enlace al canon |
| Ebook/workbook | convertir el territorio en método enseñable | cluster suficiente, promesa propia, ejercicios y revisión |
| Tool | personalizar diagnóstico o decisión | modelo explícito, evals, privacidad, utilidad y explicación |
| Producto | operar la capacidad | source of truth, arquitectura, runtime y evidencia propia |

## Reglas

1. Un ebook no es una concatenación de artículos. Requiere secuencia, síntesis, método y valor original.
2. Una tool no es repurposing. Es producto o lead magnet interactivo y necesita PDR, owner, modelo, privacidad,
   analytics, QA y ejecución formal.
3. El contenido puede crear lenguaje e hipótesis; no crea features, schemas, tasks ni claims de disponibilidad.
4. Separa evidencia científica, doctrina editorial, casos aplicados y evidencia de producto.
5. Una tool diagnóstica debe ser útil sin venta, explicar incertidumbre y evitar scores mágicos.
6. El contenido que precede un producto debe educar sobre el problema y sus decisiones, no fingir que el
   producto ya existe.
7. El aprendizaje puede viajar en ambas direcciones, pero cada capa conserva su source of truth y sus gates.

## Flujo de handoff

```text
señal editorial
  -> research/hipótesis
  -> validación empírica
  -> decisión de producto/arquitectura
  -> task formal
  -> evidencia del producto
  -> actualización editorial con límites
```

## Intake adicional

- ¿El activo siguiente resuelve un trabajo nuevo o solo cambia el formato?
- ¿Qué evidencia demuestra demanda o utilidad?
- ¿Qué debe aprender la persona antes de diagnosticar, diseñar u operar?
- ¿Qué claims pertenecen al conocimiento y cuáles requieren producto real?
- ¿Cuál es el gate que impide anunciar una capacidad inexistente?
- ¿Qué runtime, consentimiento, privacidad y medición aparecerían si hay tool?

## Caso de referencia Efeonce

Creative Workflows usa esta escalera para que la Pillar y sus satélites creen soporte científico e intelectual;
un ebook futuro convierta el cluster en método; una eventual `Creative Workflow Opportunity Mapper` ayude a
elegir el primer workflow; y Creative Studio materialice operación y escala solo cuando su producto lo autorice.
