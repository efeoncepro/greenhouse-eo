# Efeonce Operating Code V1

Estado: **Accepted**  
Fecha: 2026-07-09  
Owner: Efeonce / Greenhouse  
Scope: cultura operativa interna, hiring, onboarding, performance, rituales de trabajo y delivery

## Proposito

Este documento baja el Why de Efeonce a comportamiento observable. No es un manifiesto de valores ni copy de marca: es el codigo operativo para decidir que se contrata, que se premia, que se corrige y que se considera buen trabajo.

Why canonico:

> **No te entregamos crecimiento. Lo construimos contigo, y te dejamos mas capaz de sostenerlo.**

Regla cultural:

> **En Efeonce se valora a quien deja al cliente mas capaz, deja el sistema con mas memoria y conecta su trabajo con crecimiento real.**

## Los 5 principios

### 1. Construimos con el cliente, no para el cliente

El cliente no recibe una caja cerrada al final del proceso. Opera con nosotros, entiende las decisiones, participa temprano y ve el avance con contexto.

Comportamientos esperados:

- Involucrar al cliente antes de que una decision este demasiado cerrada.
- Explicar tradeoffs, no solo presentar conclusiones.
- Mostrar estado, riesgos y bloqueos sin maquillaje.
- Convertir aprobaciones en aprendizaje, no en tramite.

Senales de drift:

- El cliente solo ve el trabajo cuando ya esta listo.
- Se esconden riesgos para "no preocupar".
- Se confunde co-creacion con pedir feedback tarde.

### 2. Ensenamos porque somos expertos

Un experto que no ensena vuelve dependiente al cliente. Efeonce debe dejar mas criterio instalado en cada ciclo.

Comportamientos esperados:

- Explicar por que una recomendacion es correcta.
- Crear frameworks, checklists o ejemplos reutilizables.
- Corregir briefs pobres con criterio, no solo ejecutarlos.
- Hacer que el cliente tome mejores decisiones despues de trabajar con nosotros.

Senales de drift:

- "El cliente no entiende" se usa como excusa.
- El conocimiento queda en la cabeza de una persona.
- La recomendacion depende de autoridad, no de evidencia.

### 3. Dejamos memoria, no solo entregables

Cada ciclo debe dejar historial, decisiones, datos, aprendizajes y activos que hagan mas inteligente el siguiente ciclo.

Comportamientos esperados:

- Documentar decisiones relevantes y su razon.
- Registrar aprendizajes en Greenhouse, Kortex, Verk o el sistema canonico que corresponda.
- Reusar patrones antes de reinventar.
- Convertir trabajo repetido en proceso, template, primitive, playbook o automation.

Senales de drift:

- Cada proyecto empieza de cero.
- El aprendizaje vive en chats sueltos.
- Se entrega una pieza sin dejar contexto operativo.

### 4. Medimos impacto, no actividad

La actividad no es crecimiento. El trabajo debe acercarse a negocio: pipeline, revenue, NRR, eficiencia operativa, time-to-market o calidad de decision.

Comportamientos esperados:

- Distinguir output de outcome.
- Conectar entregables con Revenue Enabled cuando sea posible.
- Usar ICO para explicar eficiencia, calidad y aprendizaje.
- Rechazar vanity metrics como argumento principal.

Senales de drift:

- Se reporta volumen sin impacto.
- Se celebra estar ocupado como si fuera avanzar.
- Una metrica se muestra porque se ve bien, no porque decide algo.

### 5. Operamos como un sistema, no como silos

Efeonce compite como Growth Operating System. Estrategia, creatividad, medios, datos y software deben trabajar como una sola operacion.

Comportamientos esperados:

- Pensar en el efecto de una decision sobre otras capabilities.
- Mantener contexto compartido entre equipos.
- Elevar problemas de sistema, no solo resolver el sintoma local.
- Usar Greenhouse/Kortex/Verk como backbone, no como decoracion.

Senales de drift:

- Cada unidad optimiza su parte y rompe el todo.
- El cliente repite contexto a varios equipos.
- Una entrega no conversa con datos, CRM, distribucion o operacion.

## Preguntas obligatorias por ritual

### Kickoff

- Que debe aprender el cliente en este ciclo?
- Que decision podra tomar mejor despues de trabajar con nosotros?
- Que memoria debe quedar registrada?
- Que dato conectara este trabajo con negocio?
- Que parte del sistema debe mejorar, no solo completarse?

### Weekly / seguimiento

- Que riesgo estamos viendo antes de que explote?
- Que necesita entender el cliente para decidir mejor?
- Que bloqueo es de sistema y no de una persona?
- Que estamos aprendiendo que debe quedar registrado?

### Cierre / retrospective

- Que quedo mas claro para el cliente?
- Que quedo documentado o reutilizable?
- Que dato acumulamos?
- Que se puede convertir en proceso, template o automation?
- Que hariamos mejor en el proximo ciclo?

## Hiring: como se evalua

No contratar por "culture fit" como intuicion. Evaluar conductas observables alineadas al Operating Code.

Senales positivas:

- Ensena con claridad sin ponerse por encima.
- Documenta decisiones y aprendizajes.
- Trabaja con transparencia bajo presion.
- Piensa en sistemas, no solo en tareas.
- Conecta su craft con impacto de negocio.

Senales de alerta:

- Retiene conocimiento como poder.
- Necesita opacidad para verse indispensable.
- Solo valora la ejecucion visible, no la memoria que deja.
- Habla de impacto sin evidencia.
- Culpa al cliente por no entender, en vez de elevar su criterio.

## Onboarding: dia 0 a dia 30

Toda persona nueva debe entender que Efeonce no es una fabrica de entregables. El onboarding debe cubrir:

1. Why + Operating Code.
2. Growth Operating System / ASaaS.
3. Greenhouse, Kortex y Verk como prueba del sistema.
4. Loop Marketing + ICO como metodo.
5. Revenue Enabled como criterio de impacto.
6. Casos reales y aprendizajes reutilizables.
7. Ritual de documentar memoria desde el primer proyecto.

## Performance: que se premia

La evaluacion de desempeno debe mirar resultados y conducta cultural.

Dimensiones:

- **Transparencia:** muestra estado, riesgos y decisiones con contexto.
- **Educacion:** eleva criterio del cliente y del equipo.
- **Memoria:** deja aprendizajes, datos o activos reutilizables.
- **Impacto:** conecta el trabajo con una metrica de negocio o decision.
- **Sistema:** reduce fragmentacion y mejora el backbone operativo.

No premiar como cultura:

- Heroismo que compensa mala operacion.
- Velocidad que deja deuda invisible.
- Opacidad que vuelve a alguien "indispensable".
- Mucho output sin aprendizaje acumulado.

## Lenguaje interno

Frases que deben volverse normales:

- "Esto deja memoria o solo entrega?"
- "El cliente queda mas capaz despues de esto?"
- "Donde queda registrado?"
- "Que dato prueba que esto movio negocio?"
- "Que parte del sistema mejora con este trabajo?"
- "No escondamos el problema; mostremoslo con contexto."

## Relacion con Greenhouse

Greenhouse debe materializar el Operating Code:

- Transparencia: estado vivo, riesgos, decisiones y bloqueos visibles.
- Co-creacion: aprobaciones, briefs, solicitudes y feedback dentro del portal.
- Educacion: insights, explicaciones, frameworks y Nexa como coach gobernado.
- Memoria: Account 360, Person 360, historial ICO, proyectos y aprendizajes.
- Impacto: Revenue Enabled y Pulse conectando operacion con expansion/retencion.

Si Greenhouse solo muestra metricas, cumple una parte del Why. Si permite actuar, aprender y acumular criterio, lo vuelve cultura.
