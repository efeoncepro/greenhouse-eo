## Por quГ© existe este doc

El desarrollo agГ©ntico deja **deuda cognitiva**: una feature se implementa detrГЎs de un flag `*_ENABLED` default-OFF, pasa todos los gates, se mergeaвЂ¦ y **el flag nunca se prende** en staging/prod porque nadie lo registrГі en un lugar encontrable. Este ledger es ese lugar: un vistazo Гєnico de **quГ© flags estГЎn prendidos dГіnde** y **quГ© queda pendiente de prender**.

Es **encontrable** desde: `CLAUDE.md` (Runtime Rollout Completion Gate), `AGENTS.md`, y el Г­ndice de `docs/operations/`.

## Dos mecanismos de flags (no confundir)

|---|---|---|---|
| **Env-var flags** (`*_ENABLED`) вЂ” **este doc** | Vercel env vars (por environment) + `.env.local` | `vercel env add` + **redeploy** | Casi todo: features de runtime, integraciones, rollouts graduales sin tabla |

## вљ пёЏ Reglas duras

- **NUNCA** declarar un env-var flag nuevo (`*_ENABLED`) sin agregar una fila a **В§ Inventario** y, si queda code-complete pero sin prender, a **В§ Pendientes de acciГіn** вЂ” en el mismo PR. **Enforcement mecГЎnico:** `pnpm docs:closure-check` corre `feature-flags-audit --strict` y **falla (exit 1) si hay un flag en cГіdigo sin registrar acГЎ** в†’ ningГєn cierre/closure pasa con un flag sin documentar.
- **NUNCA** considerar un flag "rolled out" hasta verlo en el environment correcto (`vercel env ls`) **+ redeploy aplicado**. `code complete` в‰  `operationally complete` (ver Runtime Rollout Completion Gate en `CLAUDE.md`).
- **NUNCA prender un flag sГіlo en Vercel.** Un `*_ENABLED` se lee en **cada runtime que ejecuta el cГіdigo gateado**, y Greenhouse tiene 5 runtimes con env vars independientes: **Vercel** (app Next.js) y 4 servicios **Cloud Run** (`ops-worker`, `commercial-cost-worker`, `ico-batch-worker`, `hubspot-greenhouse-integration`). Prenderlo en uno NO lo prende en los otros. **SIEMPRE** ejecutar el paso 0 de В§`CГіmo prender un env-var flag` (mapear runtimes ANTES de prender) y declarar en la fila del ledger **en quГ© runtime(s) vive**. Un flag que gatea una **projection reactiva / consumer / cron** corre en el **ops-worker**, NO en Vercel вЂ” prenderlo en Vercel no hace absolutamente nada. Caso fuente 2026-07-09: `GROWTH_EBOOK_EMAIL_DELIVERY_ENABLED` (email de entrega del ebook) vive sГіlo en el `ops-worker`; el runbook de este doc sГіlo enseГ±aba `vercel env add`, y prenderlo ahГ­ habrГ­a dejado el email muerto con la success card prometiГ©ndoselo al usuario.
- **NUNCA prender un flag de Cloud Run sГіlo con `gcloud run services update --update-env-vars`.** Los `deploy.sh` de los workers usan **`--set-env-vars` (destructivo)**: reescriben el set completo, asГ­ que una var agregada out-of-band **desaparece en el siguiente deploy, en silencio**. El SoT es **`services/<worker>/deploy.sh`** (declarar el flag ahГ­) + aplicar en vivo para efecto inmediato. **Los dos pasos, siempre.** Caso fuente 2026-07-10: el flag del ebook se prendiГі en la revisiГіn `00470`, la `00473` lo borrГі, y el consumer reactivo registrГі `skip: flag OFF` вЂ” el ledger decГ­a ON y la realidad era OFF. Un flag "prendido" que no estГЎ declarado en `deploy.sh` es una bomba de tiempo, no un rollout.
- **SIEMPRE** que prendas/apagues un flag en un environment, actualizГЎ la **В§ Snapshot** (con fecha) y, si cerrГі un pendiente, sacГЎ la fila de **В§ Pendientes de acciГіn**.
- **NUNCA** confГ­es en este doc como verdad live para una decisiГіn crГ­tica вЂ” la **verdad live es `vercel env ls`**. Este doc es el ledger humano (intenciГіn + pendientes + Гєltimo snapshot conocido).
- Para flags `NEXT_PUBLIC_*`: se hornean en el bundle **en build time** в†’ prenderlos requiere un **build fresco** (push o redeploy con build cache desmarcado), no un redeploy que reusa build.

## В§ CГіmo prender un env-var flag

> вљ пёЏ **Un flag no es "el flag de Vercel".** Es una env var leГ­da por **cada runtime que ejecuta el cГіdigo gateado**. Greenhouse tiene **5 runtimes con env vars totalmente independientes**. Prenderlo en el runtime equivocado = no pasa nada (y peor: parece prendido en el ledger). **El paso 0 no es opcional.**

### Los 5 runtimes y sus env vars

| Runtime | QuГ© ejecuta | CГіmo se prende | CГіmo se verifica |
|---|---|---|---|
| **Vercel** (app Next.js) | API routes, server components, UI, crons de `vercel.json` | `vercel env add` + redeploy | `vercel env ls` |
| **`ops-worker`** (Cloud Run, `us-east4`) | **projections reactivas, consumers del outbox, crons de Cloud Scheduler**, emails async | `gcloud run services update ops-worker` | `gcloud run revisions describe` |
| **`commercial-cost-worker`** (Cloud Run, `us-east4`) | jobs de costo comercial | `gcloud run services update commercial-cost-worker` | idem |
| **`ico-batch-worker`** (Cloud Run, `us-east4`) | batches ICO / delivery metrics | `gcloud run services update ico-batch-worker` | idem |
| **`hubspot-greenhouse-integration`** (Cloud Run, `us-central1`) | bridge + webhooks HubSpot | `gcloud run services update hubspot-greenhouse-integration --region us-central1` | idem |

### Paso 0 вЂ” Mapear en QUГ‰ runtime(s) vive el flag (ANTES de prender)

```bash
# Encontrar dГіnde se LEE el flag (no dГіnde se declara)
grep -rn "<FLAG_NAME>\|is<Feature>Enabled" src/ services/ | grep -v __tests__
```

TraducГ­ cada path a su runtime:

| Path donde se lee | Runtime real |
|---|---|
| `src/app/**`, `src/views/**`, `src/components/**` | **Vercel** |
| `src/lib/sync/projections/**` (projection reactiva) | **ops-worker** (la drena `ops-reactive-*`) |
| `services/ops-worker/**` | **ops-worker** |
| `services/<otro-worker>/**` | ese worker |
| `src/lib/**` compartido | **depende de los callers** в†’ seguir la cadena hasta un `src/app/**` (Vercel) y/o una projection/worker (Cloud Run). Puede ser **los dos**. |

**HeurГ­stica de bolsillo:** Вїel flag gatea algo **async** (email de respaldo, projection, consumer del outbox, cron de Scheduler, materializer)? в†’ **ops-worker, NO Vercel.** ВїGatea una **ruta/superficie visible**? в†’ **Vercel.** ВїAmbas cosas? в†’ **ambos** (ej. `GROWTH_AI_VISIBILITY_REPORT_EMAIL_ENABLED`, marcado `(ops-worker + Vercel)` en el snapshot).

### Paso 1 вЂ” Prender en CADA runtime del mapeo

> вљ пёЏ **En Cloud Run, `--update-env-vars` NO es el flip: es un parche efГ­mero.** Los `deploy.sh` de los workers usan **`--set-env-vars` (destructivo)**, que reescribe el set completo de env vars. Una var agregada out-of-band con `--update-env-vars` **sobrevive hasta el siguiente deploy del worker y despuГ©s desaparece en silencio**. El **SoT es `services/<worker>/deploy.sh`**: ahГ­ se declara el flag. Por eso ese archivo estГЎ lleno de comentarios "Declarativo acГЎ para que `--set-env-vars` (destructivo) NO lo borre".

```bash
# --- Vercel (si aplica) --------------------------------------------------
printf %s "true" | vercel env add <FLAG_NAME> <environment> --scope efeonce-7670142f
#    environments: Production | staging | preview | development  (staging = custom env)
#    Si el flag tiene mirror NEXT_PUBLIC_*, agregar AMBAS.
```

**Cloud Run (si aplica) вЂ” SIEMPRE los dos pasos, en este orden:**

**(a) Declararlo en `services/<worker>/deploy.sh`** (SoT, sobrevive deploys) siguiendo la convenciГіn del archivo:

```bash
# <TASK-###> вЂ” <quГ© gatea>. Declarativo acГЎ para que `--set-env-vars` (destructivo) NO lo
# borre en cada redeploy. Rollback (<5min): gcloud run services update <svc> --update-env-vars <FLAG>=false
<FLAG_NAME>="${<FLAG_NAME>:-true}"
ENV_VARS="${ENV_VARS},<FLAG_NAME>=${<FLAG_NAME>}"
```

**(b) Aplicarlo en vivo** para que tome efecto ya, sin esperar un deploy del worker:

```bash
gcloud run services update <servicio> \
  --region <us-east4|us-central1> --project efeonce-group \
  --update-env-vars <FLAG_NAME>=true
#    Crea la revisiГіn y enruta trГЎfico. Pero SIN el paso (a), el prГіximo deploy lo borra.
```

Hacer sГіlo (b) es el bug class del 2026-07-10: `GROWTH_EBOOK_EMAIL_DELIVERY_ENABLED` se prendiГі en la revisiГіn `00470`, tres deploys despuГ©s (`00473`) ya no estaba, y el consumer reactivo registrГі `skip: flag OFF` mientras la success card del form le prometГ­a el email al usuario. **Fallo silencioso: el ledger decГ­a ON, la realidad era OFF.**

### Paso 2 вЂ” Redeploy donde haga falta

- **Vercel server var** (`*_ENABLED`): aplica en el prГіximo deploy (las env vars NO se toman en caliente).
- **Vercel `NEXT_PUBLIC_*`**: se hornea en build в†’ requiere **BUILD FRESCO** (push a la rama del env, o Redeploy con "Use existing Build Cache" DESMARCADO).
- **Cloud Run**: el `services update` ya crea la revisiГіn. Nada mГЎs.

### Paso 3 вЂ” Verificar en CADA runtime (que la var exista Y que el consumer se comporte)

```bash
# Vercel
vercel env ls | grep <FLAG_NAME>

# Cloud Run вЂ” leer la env de la REVISIГ“N ACTIVA (no del servicio "en general")
REV=$(gcloud run services describe <servicio> --region <region> --project efeonce-group \
        --format='value(status.latestReadyRevisionName)')
gcloud run revisions describe "$REV" --region <region> --project efeonce-group \
  --format="json(spec.containers[0].env)" | grep -A1 <FLAG_NAME>
```

Y ademГЎs: **ejercitar el flujo real**. Que la var exista no prueba que el consumer haga lo suyo. Si el flag gatea un email async, el smoke es "llega el email", no "la var estГЎ".

### Paso 4 вЂ” Actualizar este ledger

Registrar en la fila **el/los runtime(s)** donde quedГі prendido + fecha + revisiГіn (Cloud Run) вЂ” igual que
`GROWTH_EBOOK_EMAIL_DELIVERY_ENABLED` (**ops-worker**, rev `ops-worker-00470-898`). Sin el runtime explГ­cito, el prГіximo agente asume Vercel y se equivoca.

### Apagar (rollback)

```bash
vercel env rm <FLAG_NAME> <environment> --scope efeonce-7670142f   # + redeploy
gcloud run services update <servicio> --region <region> --project efeonce-group \
  --remove-env-vars <FLAG_NAME>
```

(Quitar la var = OFF/default.) **Apagar tambiГ©n es multi-runtime**: quitarla de Vercel no la apaga en el worker.

Para los **PG rollout flags** (`home_rollout_flags`): se prenden vГ­a admin endpoint sin redeploy вЂ” ver el platform doc.
