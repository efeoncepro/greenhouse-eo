## Outcome

The Product Library failure is two independent OAuth defects, not a Product entitlement or deprecated-scope problem:

1. HubSpot accepted each authorization request but rejected all three authorization grants with `Please provide a valid recaptcha value`.
2. A fresh authorization URL generated by the deployed Kortex control plane omitted the now-required `crm.objects.products.read` scope, even though HubSpot Developer Project build `#13` contains it as required.

The active ANAM installation remained intact throughout diagnosis and consent. It now carries 110 scopes, including `crm.objects.products.read`.


This establishes that the prior requests reached HubSpot and failed at the human grant step before token exchange/callback. It rules out callback processing as the cause of those three attempts.

## Runtime configuration drift

The authorization endpoint:

```text
https://kortex-control-plane-758246035804.us-central1.run.app/hubspot/oauth/authorize?hubspot_portal_id=19893546&lock_hubspot_portal=true
```

generated a HubSpot consent request without `crm.objects.products.read`. HubSpot rejected that request before consent with a missing-required-scope error. Manually adding only `crm.objects.products.read` produced a valid consent page locked to ANAM and showing the incremental permission to read Product Library records.

Therefore the deployed control plane is not yet aligned with the deployed HubSpot app manifest. The durable remediation belongs in Kortex executable runtime: rebuild/redeploy the control plane from the current manifest/configuration, then verify the generated URL contains all required scopes. Greenhouse keeps only this canonical diagnosis.

See the official [scopes documentation](https://developers.hubspot.com/docs/apps/developer-platform/build-apps/authentication/scopes) and [OAuth guide](https://developers.hubspot.com/docs/apps/developer-platform/build-apps/authentication/oauth/working-with-oauth).
