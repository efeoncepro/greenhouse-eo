# Issue Tracker

Pipeline de incidentes operativos del portal.

## ConvenciГіn

- Los issues usan `ISSUE-###` como ID estable.
- Los archivos viven en `docs/issues/{open,resolved}/`.
- Un issue se mueve a `resolved/` cuando se confirma la soluciГіn en el ambiente afectado.
- A diferencia de las tasks (`TASK-###`), los issues son reactivos: documentan un problema encontrado en runtime, no trabajo planificado.

## Estados

- `open`: incidente detectado, en diagnГіstico o pendiente de soluciГіn.
- `resolved`: soluciГіn aplicada y verificada en el ambiente afectado.

## Protocolo operativo

- El modelo operativo canГіnico de issues vive en `docs/operations/ISSUE_OPERATING_MODEL_V1.md`.
- Un issue puede resolverse directamente sin task si el fix es localizado y verificable.
- Al resolverse, el archivo debe moverse de `docs/issues/open/` a `docs/issues/resolved/` y el tracker debe actualizarse en el mismo lote.

## Plantilla

```markdown
# ISSUE-### вЂ” TГ­tulo breve

## Ambiente

production | staging | preview

## Detectado

Fecha, canal de detecciГіn (Slack alert, Sentry, Admin Center, usuario)

## SГ­ntoma

QuГ© se observa desde fuera.

## Causa raГ­z

QuГ© lo provoca a nivel tГ©cnico.

## Impacto

QuГ© funcionalidad estГЎ afectada y para quiГ©n.

## SoluciГіn

QuГ© se hizo o hay que hacer para resolverlo.

## VerificaciГіn

CГіmo confirmar que se resolviГі.

## Estado

open | resolved

## Relacionado

Tasks, docs de arquitectura, o commits relacionados.
```
