Perfiles de pago — donde viven y por qué

Los perfiles de pago de personas viven dentro de su Person 360. Los de accionistas viven dentro de Shareholder 360. La surface centralizada en Finance solo existe para operaciones cross-entity: cola de aprobación, drift cuando faltan perfiles para obligaciones vivas, y audit cross-cutting. Una sola pieza de UI reutilizable, tres mounts naturales.

Cómo se construye sin parche

Pieza reutilizable

  • <PaymentProfilesPanel> — el componente real con CRUD. Recibe beneficiaryType + beneficiaryId.
  • Modo embedded: sin header propio, sin filtros (lo provee el host: Person 360 o Shareholder 360).
  • Modo standalone: dialog flotante o página dedicada (en V2 si hace falta).

Surface ops sin duplicación

  • Cola: GET /api/admin/finance/payment-profiles?status=pending_approval
  • Drift: nuevo helper getPaymentProfileDriftForActiveObligations() + endpoint /queue
  • Tabla universal: solo lectura. Click → deep link al 360 correcto.
  • Sin botón "Crear": la creación es contextual al beneficiario.

Resiliencia

  • Si el panel falla, Person 360 sigue funcionando (Error Boundary).
  • Si user no tiene capability finance.payment_profiles.read, el tab "Pago" no se muestra.
  • El backend ya devuelve enmascarado por default; revelar requiere capability + motivo + audit.

Escalabilidad

  • Agregar provider nuevo = INSERT en payment_provider_catalog. Cero código UI.
  • Agregar nuevo beneficiary type (V2 supplier) = montar <PaymentProfilesPanel> en su 360. Cero refactor.
  • Eventos outbox listos para Notification Hub (TASK-716) y reactive projections.
