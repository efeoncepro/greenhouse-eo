TASK-758 — Mockup recibos por régimen (post-fix)

Contrato visual canonizado. 4 cards = 4 regímenes resueltos por el helper único resolveReceiptRegime(entry). El preview MUI y el PDF descargable renderizan este mismo layout, byte-by-byte estructural — son consumers declarativos del mismo buildReceiptPresentation().

Cada card incluye al final un bloque "Cambios vs hoy (v3)" que diferencia añadido, eliminado y renombrado.

Template: RECEIPT_TEMPLATE_VERSION bumpa '3' → '4'. Cache GCS regenera lazy en próximo acceso.

Notas de implementación (skills aplicadas)

  • Tokens (V1 + DESIGN.md): tipografía Geist Sans body + Poppins display (max 2 familias); font-variant-numeric: tabular-nums para todos los números — NUNCA monospace en surfaces user-facing; spacing scale {0.5, 1, 1.5, 2, 3, 4, 5, 6, 8, 10, 12}; customBorderRadius.md = 6 (cards), .sm = 4 (info-blocks).
  • Color (semántica V1): info/warning/error reservados para STATES (info-block azul = aclaración, warning amarillo = adjustment, error rojo = excluded). --success-text: #2E7D32 es la variante contrast-safe del lime #6ec207 (que falla 4.5:1 sobre blanco — solo apto para chips/borders).
  • A11y (WCAG 2.2 AA): cada <article> recibe aria-label con régimen + nombre para anuncio único de screen reader. InfoBlock variants (info/warning/error) NO comunican estado solo por color — siempre llevan título + body explicativo. Diff annotations llevan tokens semánticos de texto (línea tachada para "removed", color para refuerzo, NO color como única señal).
  • Motion (greenhouse-microinteractions): el recibo en sí es estático — display, no interactivo. EN PRODUCCIÓN, el PayrollReceiptDialog que envuelve este preview debe: (a) mostrar Skeleton MUI cuando el PDF se está regenerando por mismatch de RECEIPT_TEMPLATE_VERSION; (b) role="status" + aria-live="polite" en el contenedor que muestra "Regenerando recibo…"; (c) respetar prefers-reduced-motion en el dialog open/close (200ms decelerado canónico, sin transform de gran amplitud).
  • Microinteracciones del receipt: ninguna intrínseca al recibo. Botones de acción ("Descargar PDF", "Reenviar email") viven fuera del recibo — son responsabilidad del Dialog/Toolbar que lo monta, no de este contrato.
  • Loading lazy: primer acceso post-bump v3→v4 dispara regen (~2s en server). UI debe mostrar <Skeleton variant="rectangular" height={500} /> + texto "Regenerando recibo con plantilla actualizada…" en role="status". NO usar spinner suelto (rompe layout).
