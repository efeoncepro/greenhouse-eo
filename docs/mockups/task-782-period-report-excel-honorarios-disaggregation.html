Layout decision: dos secciones dentro de una sola pestaña Chile

El operador HR / compliance lee el mes calendario completo de pago Chile en una sola pestaña, con secciones internas separadas para dependientes y honorarios. Esto preserva la lectura unificada por mes (use case principal del operador) y mantiene la disaggregación auditable. Si compliance pidiera pestañas separadas más adelante, el refactor es trivial — la separación viene del helper canónico, no del layout. Alternativa rechazada: pestañas separadas Chile - Dependientes y Chile - Honorarios — fragmenta lectura mensual sin ganar nada operativo.

Pestaña Internacional — paridad estructural

NO modelada en este mockup por simplicidad visual. Sigue el mismo patrón: dos secciones internas (Internacional Deel / Internacional interno) con subtotales separados. Sin columnas de descuentos previsionales ni retención SII (no aplican). Columna informativa adicional deelContractId para la sección Deel cuando existe.

Cambios Excel vs v3

  • + Pestaña Chile dividida en 2 secciones internas (`▼ Sección 1 Chile dependiente` / `▼ Sección 2 Honorarios`) con subtotales independientes
  • + Columnas nuevas dedicadas a honorarios: Tasa SII + Retención SII
  • + Para honorarios, columnas previsionales (AFP/Salud/Cesantía/IUSC/APV) llenas con — (text-faint), NUNCA $0 (lectura ambigua)
  • + Para dependientes, columnas SII (Tasa/Retención) llenas con —
  • + Subtotal "Total descuentos previsionales" (solo dependientes) ≠ Subtotal "Total retención SII honorarios" (solo honorarios)
  • + Badge régimen color-coded en columna 3 (paridad PDF + recibos TASK-758)
  • ⇄ Subtotal único "Total descuentos Chile" → 2 subtotales independientes
  • − Mezcla AFP/salud/cesantía/IUSC con retención SII en pool indiferenciado
  • − Honorarios sin columna propia para tasa/retención

Reglas duras de implementación (vinculantes)

  1. Single source of truth para clasificación: ambos surfaces consumen resolveReceiptRegime exportado por TASK-758. Helper auxiliar groupEntriesByRegime(entries) retorna Record<ReceiptRegime, PayrollEntry[]>. NUNCA reimplementar el detector inline.
  2. Orden canónico de regímenes en ambos surfaces: chile_dependent → honorarios → international_deel → international_internal. Stable. No depende del orden alfabético del nombre.
  3. Régimen column con 4 valores (badges color-coded paridad TASK-758): CL-DEP verde · HON ámbar · DEEL mostaza · INT índigo. Los hex viven en receipt-presenter.ts y se exportan para reuso aquí.
  4. Celdas no aplicables = — con clase dim (text-faint), NUNCA $0. Honorarios en columnas previsionales = —. Dependientes en columnas SII = —. Distinción semántica: $0 = aplica pero el monto es cero; — = no aplica al régimen.
  5. Subtotales mutuamente excluyentes: "Total descuentos previsionales" suma SOLO dependientes; "Total retención SII honorarios" suma SOLO honorarios. Cualquier cross-contamination = bug.
  6. Estado terminal excluded: fila se renderiza con $0 en columna Bruto/Neto y — en todas las demás + chip warning "⚠ excluido" en el nombre. NO se omite (operador necesita contar el universo).
  7. Tokens prohibidos (heredados auditoría TASK-758):
    • NUNCA font-family: monospace en celdas user-facing.
    • NUNCA font-feature-settings: 'tnum' — usar font-variant-numeric: tabular-nums.
    • NUNCA borderRadius off-scale (3, 5, 7, 12). Cards = customBorderRadius.md (6).
    • NUNCA spacing off-scale (36, 28). Usar spacing(n) con escala canónica.
    • NUNCA color como única señal — subtotales llevan label semántico explícito.
  8. A11y mandatorio: tabla con <caption> describiendo agrupación; <th scope="col"> en headers; group divider rows con texto descriptivo + colspan completo (no solo color); aria-label en el componente raíz del PDF/Excel describiendo período + cardinalidad.
  9. Microinteracciones de la UI surrounding (PayrollPeriodView que dispara descarga):

    • Toast success: react-toastify con autoClose: 4000, link al archivo

    • Respetar useReducedMotion — sin animaciones decorativas en el flujo de export
